## 한 문단 요약

여러 고객의 과거 문의와 프로젝트 문서를 검색하는 AI 챗봇에서 질문의 고객·프로젝트·티켓 단서가 질문 재작성 과정에서 사라지는 문제를 다뤘습니다. LLM은 검색 범위 후보만 제안하고, 실제 메타데이터 대조와 filter·fallback은 코드가 결정하도록 분리했습니다. 문서 수집·색인도 파일 단위 실패 격리와 단계별 retry를 적용했습니다.

## Context

사내 기술지원용 AI 챗봇은 여러 고객의 과거 문의와 프로젝트 문서를 함께 검색합니다. 질문에 포함된 검색 주어가 재작성 과정에서 빠지면 관련 문서보다 일반적인 문서가 우선될 수 있었습니다. 반대로 불확실한 후보를 강제 필터로 적용하면 필요한 문서를 모두 제외할 수 있었습니다.

검색 품질은 검색 로직만으로 결정되지 않았습니다. 공유 문서 저장소의 파일이 제때 수집·변환·색인되지 않거나 한 파일의 실패가 전체 배치를 멈추면 같은 검색 실패가 발생했습니다.

## Problem & Constraints

- 자연어 질문의 고객·프로젝트·티켓 단서가 명시적이지 않을 수 있습니다.
- LLM이 추정한 후보를 hard filter로 바로 쓰면 필요한 문서를 놓칠 수 있습니다.
- 제한 검색 결과가 없을 때 사용자가 받을 수 있는 답변 경로가 필요합니다.
- 외부 문서 저장소는 5xx·429, 연결 실패와 파일 변환 오류가 발생할 수 있습니다.
- 고객명, 프로젝트명, 티켓 번호, 질문과 문서 원문은 공개할 수 없습니다.

## My Role

문제를 정의하고 검색 범위 schema와 책임 분리를 설계했습니다. 질문 재작성 결과에서 검색 후보를 전달하고, 코드가 후보를 정규화해 실제 메타데이터와 대조하도록 구현했습니다. 수집·색인 경로에서는 반복 탐색과 중복 변환을 줄이고, 동시 처리 수와 실패·retry 경계를 개선했습니다.

## Key Decision

LLM에는 후보 제안만 맡겼습니다. 실제 검색 조건은 코드가 검증된 메타데이터를 기준으로 결정하고, 제한 검색 결과가 없으면 기본 검색으로 돌아가게 했습니다. 잘못된 추정 하나가 전체 검색 결과를 비우지 않도록 하기 위한 결정입니다.

## Implementation

- 질문에 명시되거나 강하게 암시된 검색 범위 후보 생성
- 후보 정규화와 실제 메타데이터 대조
- 검증된 후보만 filter 또는 boost에 반영
- 제한 검색 결과가 없을 때 기본 검색으로 fallback
- 공유 문서 저장소를 한 번 탐색한 뒤 작업 분배
- 동시 처리 수 제한과 작업별 client 사용
- download, conversion, registration 실패를 파일 단위로 분리 기록
- 재시도 가능한 실패와 즉시 중단해야 하는 오류 구분

## Evidence & Result

- rewrite, retrieval, metadata와 ingestion 변경 경로의 정적 검사와 대상 테스트를 확인했습니다.
- 제한 검색 결과가 없을 때 기본 검색으로 돌아가는 경로를 추가했습니다.
- 실패 파일과 원인을 실행 결과에 남겨 파일별로 다시 실행할 수 있게 했습니다.
- 외부 문서 수집의 반복 탐색, 동시 처리와 retry 경계를 코드에서 분리했습니다.

## Boundaries

- 운영 재색인 완료를 주장하지 않습니다.
- 검색 품질 향상률, 고객 효과와 SLA 개선을 주장하지 않습니다.
- 구현 파일 수나 변경 줄 수를 사용자 품질 개선 수치로 사용하지 않습니다.
- 내부 문서 구조, 고객·프로젝트·티켓 식별 정보는 공개하지 않습니다.

## Lessons

RAG 품질은 프롬프트나 질문 재작성만으로 유지되지 않습니다. 지식 수집·색인, 구조화된 메타데이터, 검색 범위와 fallback이 같은 처리 흐름 안에서 안정적으로 연결되어야 합니다. LLM의 유연성과 규칙 기반 백엔드의 보장을 분리하면 잘못된 추정의 영향을 제한할 수 있습니다.
