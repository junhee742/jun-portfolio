## 한 문단 요약

서로 다른 두 엔터프라이즈 제품이 하나의 DB 스키마를 공유할 때 발생할 수 있는 스케줄러 테이블 충돌을 설정 기반 격리로 해결할 수 있는지 검증했습니다. 별도 DB 계정이나 제품 간 신규 API를 추가하기 전에 더 작은 변경으로 공존할 수 있는 조건을 설계하고, 로컬 등가환경과 내부 검증환경에서 단계적으로 확인했습니다.

## Context

두 제품은 각각 독립적인 작업 스케줄러를 사용하지만 인프라 제약상 하나의 DB 스키마를 공유해야 했습니다. 동일한 기본 테이블 이름을 사용하면 한 제품의 스케줄러 상태가 다른 제품과 충돌할 수 있었습니다.

## Problem & Constraints

- 별도 DB 계정을 추가하면 인프라 협업과 권한 관리 범위가 커집니다.
- 제품 간 신규 스케줄러 API를 만들면 개발 범위와 운영 책임이 복잡해집니다.
- 고객 운영환경에 적용하기 전에 공존 조건과 롤백 가능성을 내부에서 확인해야 했습니다.
- AI가 생성한 산출물과 제가 직접 수행한 판단을 구분해야 했습니다.

## My Role

충돌 원인을 분석하고 prefix 기반 격리 구조, 수용 기준, 검증 시나리오, 전환·롤백·모니터링 기준을 설계했습니다. AI coding agent가 이 기준을 바탕으로 DDL 변형, 2-node 테스트 도구, 전환·모니터링·롤백 자산을 만들고 검증을 실행하도록 오케스트레이션했습니다. 이후 결과를 검토하고 기술 방향을 판단했습니다.

구현 산출물을 제가 직접 코딩하거나 검증 명령을 실행한 것으로 표현하지 않습니다. 설계·검증 기준·결과 판정과 조율은 제가 담당했고, 산출물 생성과 검증 실행은 AI coding agent가 담당했습니다.

## Key Decision

별도 인프라를 추가하거나 제품 간 결합을 만드는 대신, 스케줄러가 사용하는 테이블 prefix를 제품별로 분리하는 설정 기반 접근을 우선 검증했습니다. 변경 범위를 줄이고 기존 실행 모델을 유지하면서 충돌 지점만 격리할 수 있기 때문입니다.

## Implementation

- 제품별 스케줄러 테이블 prefix 분리
- 두 노드가 같은 스키마를 사용하는 등가환경 구성
- 실행 슬롯의 중복·누락 여부 확인
- 한 노드 중단 시 인계 확인
- 전환 전후 확인 항목과 롤백 기준 문서화

## Evidence & Result

- 로컬 2-node 등가환경에서 21/21 실행 슬롯 exactly-once 확인
- 중복 실행 0건
- 한 노드 중단 시 1슬롯 내 인계 확인
- 내부 검증환경에서 16시간 연속 검증

위 결과는 로컬 등가환경과 내부 검증환경에서 확인한 범위입니다.

## Boundaries

- 고객 운영환경 적용 완료를 주장하지 않습니다.
- 운영 장애 감소나 비용 절감 효과를 주장하지 않습니다.
- 내부 검증을 production-proven으로 표현하지 않습니다.
- 고객·산업·내부 제품·테이블·서버·DB 식별 정보는 공개하지 않습니다.

## Lessons

엔터프라이즈 환경에서는 새로운 시스템을 추가하기 전에 기존 설정 경계만으로 문제를 격리할 수 있는지 확인할 가치가 있습니다. AI를 구현자로 사용할 때도 사람이 문제 정의, 수용 기준, 검증 범위와 결과 판정을 소유해야 합니다.
