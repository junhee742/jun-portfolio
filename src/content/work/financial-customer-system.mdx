## 한 문단 요약

금융권 고객사의 채팅상담 기능을 개선하며 상담 상태, 기간계 공휴일 정보와 고객 화면을 개발했습니다. 기능 구현 전에 내부망과 방화벽, 이중화 URL, 테스트 계정과 운영 반영 일정을 확인하고, 8개 모바일·브라우저 조합의 검증과 현업 테스트 대응, 운영 이행까지 같은 작업 범위로 관리했습니다.

## Context

금융권 구축에서는 요구사항이 정리됐더라도 개발 PC, 고객사 개발계·운영계 서버와 기간계 사이의 통신 조건이 맞지 않으면 개발과 검증을 진행할 수 없습니다. 연동 대상이나 방화벽 기안이 누락되면 기능 개발을 마친 뒤에도 현업 테스트가 멈출 수 있었습니다.

## Problem & Constraints

- 고객 개발계와 운영계가 분리되어 있고 통신 경로와 권한이 서로 다릅니다.
- 기간계는 이중화 URL, timeout, 실패 처리와 재실행 영향을 함께 확인해야 합니다.
- 고객 화면은 OS와 브라우저, 인앱 브라우저에 따라 스크롤과 복귀 동작이 달라질 수 있습니다.
- 테스트 계정과 운영 반영 일정이 개발 일정과 함께 맞아야 합니다.
- 고객명, 내부 URL, 서버·기간계 식별 정보는 공개할 수 없습니다.

## My Role

요구사항을 화면, API, 외부 연동, 데이터와 배포 조건으로 나눴습니다. 필요한 서버, 이중화 URL, 방화벽 범위와 테스트 환경을 고객 담당자와 먼저 맞춘 뒤 Java/Spring 기능을 개발했습니다. 현업 검증에서 발견된 조건을 반영하고 운영 이행까지 담당했습니다.

## Key Decision

기능 개발을 먼저 끝내고 연동 환경을 확인하지 않았습니다. 개발 전에 통신 대상, 테스트 계정, 이중화 경로와 운영 반영 절차를 확인 항목으로 두고, 구현과 고객 검증 사이에서 새로 발견되는 환경 조건을 줄이는 방향을 선택했습니다.

## Implementation

- 상담 유효 여부와 상담 해결 여부 처리
- 기간계 공휴일 정보를 제품 휴일 관리에 반영
- 중복 등록과 재실행 영향을 고려한 데이터 처리
- 고객 채팅창의 최신 메시지 복귀 기능
- 통신 대상, 이중화 URL, timeout과 실패 로그 확인
- iOS·Android와 Safari, Chrome, Naver, Kakao 인앱 브라우저 검증

## Evidence & Result

- iOS·Android와 4개 브라우저를 조합한 8개 환경에서 고객 화면을 확인했습니다.
- 요구사항 분석, 기능 개발, 고객 현업 테스트와 운영 이행을 같은 수행 범위에서 마쳤습니다.
- 연동 대상과 검증 조건을 개발 전에 정리해 기능 완료 뒤 환경 조건 때문에 테스트가 멈추는 구간을 줄였습니다.

## Boundaries

- 특정 고객명, 망 구성, 기간계 URL과 운영 식별 정보는 공개하지 않습니다.
- 이 사례의 결과를 장애 감소율이나 비용 절감 수치로 확대하지 않습니다.
- 8개 조합은 해당 고객 화면의 검증 범위이며 모든 모바일 환경의 호환성을 의미하지 않습니다.
- 운영 이행을 장기 운영 성과나 고객 만족 성과로 표현하지 않습니다.

## Lessons

고객사 구축에서 백엔드 개발은 API 구현으로 끝나지 않습니다. 권한과 망, 외부 시스템, 테스트 계정과 운영 반영 조건을 개발 전에 확인해야 코드가 고객 환경에서 실제로 동작할 수 있습니다.
