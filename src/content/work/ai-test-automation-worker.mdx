## 한 문단 요약

저장소별 규칙, 테스트 scaffold, AI 보완, 검증과 발행 경계를 연결한 테스트 자동화 worker를 구현하고 사용했습니다. 목표는 테스트를 많이 생성하는 것이 아니라 AI가 만든 변경을 검토 가능한 patch로 남기고, 검증이 실패하면 외부 branch나 PR 생성 전에 중단하는 것이었습니다.

## Context

여러 서비스의 테스트 공백을 보완하려면 서비스별 경로, mock 경계, 검증 명령과 coverage 기준을 반복해서 확인해야 했습니다. AI가 테스트를 생성하더라도 기존 사람이 작성한 파일을 침범하거나 실패한 변경이 발행되면 신뢰할 수 없었습니다.

## Problem & Constraints

- 저장소와 서비스마다 테스트 위치와 검증 명령이 다릅니다.
- 생성된 테스트가 사람 작성 파일이나 허용되지 않은 경로를 바꿀 수 있습니다.
- 생성 성공과 컴파일·실행 성공은 다릅니다.
- 검증 시간이 부족한 상태를 성공으로 처리하면 안 됩니다.
- 내부 저장소명, 서비스명, PR과 branch 식별자는 공개할 수 없습니다.

## My Role

저장소별 contract/profile, 후보 선택, scaffold 생성, AI refinement, 격리 검증, 최종 검증과 pre-publish gate를 포함한 worker 흐름을 구현하고 실행했습니다. 생성 파일 경계와 companion test 규칙을 두고, 검증 결과를 발행 여부와 분리했습니다.

## Key Decision

생성과 검증을 하나의 성공 상태로 묶지 않았습니다. AI가 변경을 만들더라도 검증을 별도 gate로 통과해야 하며, 실패하거나 기한 안에 검증이 끝나지 않으면 외부 발행을 중단하도록 했습니다.

## Implementation

- 저장소별 경로와 검증 명령을 contract/profile로 분리
- 기존 파일을 직접 덮어쓰기보다 review 가능한 patch artifact 생성
- 사람 작성 테스트와 생성 테스트의 write boundary 제한
- 생성, AI 보완, 선택 검증, 최종 검증 단계 분리
- coverage source를 read-only 입력 또는 local report로 처리
- 검증 실패 시 branch·draft PR 생성 전 중단
- 불완전한 검증을 성공으로 처리하지 않는 soft-deadline 규칙

## Evidence & Result

- 보존된 실행에서 518개 후보 중 한 대상을 선택했습니다.
- 선택된 대상은 AI refinement 1/1과 verification 1/1을 통과하고 branch 발행이 기록됐습니다.
- 한정된 단일 서비스 검증에서 CI 성공, 전체 coverage 80.7%, 755개 테스트를 확인했습니다.
- disposable clone에서 pytest가 exit code 0으로 끝났고 원본 저장소는 변경하지 않았습니다.

## Boundaries

- 80.7% coverage와 755개 테스트는 한 서비스에만 해당합니다.
- 전체 서비스의 before/after coverage 개선을 주장하지 않습니다.
- production 신뢰성이나 고객·비즈니스 효과를 주장하지 않습니다.
- auto-merge 성공을 주장하지 않습니다.
- 모든 테스트 작업이 완전 자율화됐다고 주장하지 않습니다.
- 보존된 518-to-1 실행은 특정 PR이 아니라 branch 발행까지를 증명합니다.

## Lessons

AI 자동화의 가치는 생성량보다 실패를 외부로 내보내지 않는 경계에서 나옵니다. 저장소 규칙과 검증 명령을 코드 밖의 contract로 분리하면 같은 흐름을 다른 서비스에 적용하면서도 각 저장소의 제약을 유지할 수 있습니다.
