## 어떤 문제였나

여러 서비스의 테스트 공백을 보완할 때마다 테스트 위치, 가짜 객체를 만드는 범위와 검증 명령을 다시 확인해야 했습니다. AI가 테스트 코드를 만들 수는 있지만, 기존 파일을 잘못 바꾸거나 실행되지 않는 결과를 내놓으면 그대로 사용할 수 없었습니다.

저는 테스트를 많이 생성하는 것보다, 실패한 변경을 외부로 내보내지 않는 구조를 우선했습니다.

## 제가 맡고 판단한 일

저장소마다 다른 경로와 검증 명령을 설정으로 분리하고, 대상 선택부터 테스트 초안 생성, AI 보완, 격리 검증과 브랜치 발행까지 자동화했습니다.

핵심 판단은 ‘코드를 만들었다’와 ‘실행해도 된다’를 같은 성공으로 보지 않는 것이었습니다. AI가 테스트를 만들었더라도 별도의 검증을 통과해야 다음 단계로 이동하게 했습니다. 검증에 실패하거나 정해진 시간 안에 끝나지 않으면 검토용 브랜치를 만들기 전에 멈췄습니다.

## 어떻게 구현했나

- 저장소별 테스트 위치와 검증 명령을 설정으로 분리했습니다.
- AI가 바꿀 수 있는 파일과 경로를 제한했습니다.
- 수정 내용을 바로 덮어쓰지 않고 검토할 수 있는 변경 파일로 남겼습니다.
- 원본과 분리한 임시 저장소에서 테스트를 먼저 실행했습니다.
- 테스트 결과, 코드 실행 범위와 변경 파일을 최종 확인한 뒤 검토용 브랜치를 만들었습니다.

## 결과와 확인 범위

보존된 한 차례 실행에서는 518개 후보 중 실제 대상 1개를 선택했고, AI 보완 1건과 검증 1건이 통과해 검토용 브랜치 발행까지 기록됐습니다. 별도의 한 서비스 전체 검증에서는 자동 검증(CI) 성공, 테스트가 실행한 코드 비율 80.7%와 테스트 755개를 확인했습니다.

518개 후보를 다룬 실행과 코드 실행 범위 80.7%를 확인한 서비스 검증은 서로 다른 범위입니다. 모든 서비스가 같은 수준에 도달했거나 테스트 작업 전체를 자동화한 것은 아닙니다. 내부 저장소와 브랜치 정보는 공개 자료에서 제외했습니다.
