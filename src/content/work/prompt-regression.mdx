## 어떤 문제였나

모델이 바뀐 뒤 특정 질문에 잘못 답하는 문제가 반복됐습니다. 질문 하나에 맞춰 답변 규칙만 고치면, 표현이 조금 다른 질문이나 이전까지 정상적으로 답하던 질문이 새로 깨질 수 있었습니다.

저는 접수된 오류를 일회성 수정 요청으로 보지 않았습니다. 다음 모델이나 답변 규칙 변경에서도 같은 문제를 다시 확인할 수 있도록, 질문과 기대 답변을 검증 데이터로 남겼습니다.

## 제가 맡고 판단한 일

반복된 질문과 기대 답변을 유형별로 정리하고, 같은 입력을 다시 실행하는 도구와 결과 보고서를 만들었습니다. 실패한 질문만 모으지 않고 다음 세 범위를 나눠 관리했습니다.

- 실제로 실패한 질문
- 뜻은 같지만 표현을 바꾼 질문
- 변경 전까지 정상적으로 답하던 질문

또한 답변을 만드는 과정과 질문 종류를 분류하는 내부 경로를 따로 실행했습니다. 오류가 답변 규칙에서 생겼는지, 질문 분류에서 생겼는지 빠르게 좁히기 위한 선택이었습니다.

## 어떻게 구현했나

- 질문, 기대 답변과 실패 유형을 한 묶음으로 관리했습니다.
- 같은 뜻의 다른 표현과 기존 정상 답변을 별도 검증 범위로 두었습니다.
- 답변 API와 질문 분류 경로를 나눠 실행했습니다.
- 답변 규칙과 검증 입력의 변경 이력을 함께 남겼습니다.
- 수정할 때마다 같은 검증 범위를 다시 실행하는 절차를 문서화했습니다.

## 결과와 확인 범위

보고된 실패 질문, 표현을 바꾼 질문, 기존 정상 답변과 질문 분류 사례를 함께 재실행했습니다. 답변 생성과 질문 분류를 따로 확인해 실패 지점을 좁혔고, 고객 환경과 맞춘 내부 검증환경에서 결과를 확인한 뒤 변경을 반영했습니다.

이 결과는 백엔드 API와 질문 분류 경로의 내부 검증입니다. 전체 브라우저 화면이나 고객 운영환경의 장기 성과까지 확인했다는 뜻은 아닙니다. 고객 질문, 답변 규칙 원문과 내부 주소는 공개 자료에서 제외했습니다.
