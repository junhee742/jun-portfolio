## 한 문단 요약

모델 업그레이드 뒤 반복된 AI 챗봇 답변 오류를 건별 프롬프트 수정으로 끝내지 않고 재실행 가능한 회귀 골든데이터셋으로 관리했습니다. 실패 질문, 의미가 같은 변형, 기존 정상 답변과 라우팅 사례를 함께 확인해 한 답을 고치는 과정에서 이미 동작하던 답변이 깨지는 위험을 줄였습니다.

## Context

고객이 사용하는 AI 챗봇에서 모델 변경 뒤 특정 질문의 답변 오류가 발생했고 비슷한 유형의 지원 요청이 반복되었습니다. 질문 하나에 맞춘 프롬프트 수정은 다른 표현이나 기존 정상 답변에 회귀를 만들 수 있었습니다.

## Problem & Constraints

- LLM 결과가 바뀌면 같은 질문도 다른 답을 만들 수 있습니다.
- 실패 질문만 확인하면 기존 정상 답변의 회귀를 놓칠 수 있습니다.
- 답변 생성 실패와 질문 라우팅 실패를 분리해서 확인해야 했습니다.
- 고객 질문, 프롬프트 원문과 내부 실행 경로는 공개할 수 없습니다.

## My Role

반복 지원 요청의 질문과 기대 답변을 분류하고, 평가 데이터와 반복 실행 도구·결과 리포트를 구성했습니다. 실패 패턴에 맞춰 프롬프트와 응답 규칙을 조정하고, 답변 API와 direct-router를 서로 다른 검증 경로로 나눴습니다.

## Key Decision

운영에서 발견된 실패를 일회성 수정 요청이 아니라 다음 변경을 검증하는 데이터로 취급했습니다. 실제 실패 원문만이 아니라 의미 변형, 기존 정상 답변, 라우팅 사례를 별도 평가군으로 유지했습니다.

## Implementation

- 반복 답변 오류 질문과 기대 답변을 회귀 골든데이터셋으로 구성
- 동일 의도의 표현 변형을 별도 평가군으로 관리
- 변경 전 정상 답변을 회귀 확인 범위에 포함
- 답변 API와 질문 라우팅을 분리 실행
- 프롬프트와 회귀 입력을 버전이 있는 근거로 보존
- 변경 때마다 동일한 입력을 다시 실행하는 절차 작성

## Evidence & Result

- 고정 원문은 초기 44/54에서 변경 후 54/54를 확인했습니다.
- 의미 변형 14/14와 기존 정상 답변 8/8을 확인했습니다.
- direct-router 평가군 16/16을 별도 실행 경로에서 확인했습니다.
- 고객 환경과 정렬된 내부 검증환경에서 확인한 뒤 변경을 반영했습니다.

각 평가군은 서로 다른 목적과 실행 경로를 가지므로 하나의 전체 통과 수치처럼 합치지 않습니다.

## Boundaries

- 기록된 결과는 백엔드 API와 direct-router 기준이며 전체 브라우저 회귀검증이 아닙니다.
- 고객 production 배포와 장기 운영 효과를 주장하지 않습니다.
- 시간 절감이나 비즈니스 효과를 수치로 주장하지 않습니다.
- 고객 질문·프롬프트·내부 URL·저장 구조를 공개하지 않습니다.

## Lessons

LLM 품질은 좋은 프롬프트 한 번보다 같은 기준으로 다시 확인할 수 있는 구조에서 관리됩니다. 실패 사례를 버리지 않고 데이터로 남기면 다음 모델과 프롬프트 변경의 안전망이 됩니다.
